## 2. 章节 3.5 替代方案 —— Operator + CRD 路线

- **背景**：当前实现选择 Dispatcher 容器路线。
- **待确认**：
  - 远期是否演进到 Kubernetes Operator + 自定义 CRD？
  - CCI 2.0 是否支持自定义 CRD 与 Operator 部署？需要 CCI 平台团队确认。

### 确认结果

CCI2.0不支持自定义CRD和Operator部署，因此可以将本方案改为对 Kubernetes 原生 `CronJob` 的单租租户面简化实现。

## 3. 章节 4.5.1 指标设计

- **背景**：当前实现没有 Prometheus / OpenTelemetry 指标埋点。
- **待确认**：
  - 是否需要补充以下指标：任务触发次数、Pod 创建成功/失败率、GC 删除数量、ccictl 调用耗时分布、配置热加载次数？
  - 指标暴露方案（HTTP `/metrics` 端点、推送到 AOM、写入 LTS）？
- **责任人**：SRE / 监控团队

### 确认结果

- 可以增加上面的指标作为可观测性指标，通过 Prometheus 暴露出来，并在 LTS 中写入。
- 可观测性的内容作为一个低优先级的方案，可以在后续的迭代中实现。

## 4. 章节 6.4 冗余设计 —— 多副本 HA

- **背景**：当前 `replicas: 1`，主进程或 crond 异常虽可自愈，但 Pod 重建期间会错过触发窗口。
- **待确认**：
  - 是否需要支持多副本 + leader-election（推荐 ConfigMap-based Lease，因为 CCI 2.0 是否支持 `coordination.k8s.io/v1.Lease` 待确认）？
  - HA 是否为本期 must-have？
- **责任人**：架构师 / 业务方

### 确认结果

可以增加上面的多副本和leader-election，并在后续的迭代中实现。CCI2.0不支持`coordination.k8s.io/v1.Lease`
